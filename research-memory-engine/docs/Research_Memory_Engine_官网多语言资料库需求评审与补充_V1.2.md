# Research Memory Engine 官网多语言资料库需求评审与补充

> 基于文档：`Research_Memory_Engine_官网多语言资料库特性内容变更开发文档_V1.1.md`  
> 版本：V1.2（评审补充稿）  
> 日期：2026-07-13  
> 范围：官网多语言资料库自适应处理、中文索引特性、客户端下载  
> 结论：**有条件通过，补齐本稿 P0 条目后进入开发与验收。**

## 1. 评审结论

V1.1 已覆盖价值主张、首页模块、双语文案、平台识别、下载弹窗、SEO、埋点及基础验收，方向正确。但“营销展示”和“真实产品能力”仍有边界不清的问题，以下缺口会直接影响上线真实性与可验收性：

1. “自适应处理”缺少输入分类、策略优先级、失败降级、重试与用户可见状态。
2. “中文索引”缺少索引对象、字段来源、原文与派生内容隔离、增量更新和删除一致性。
3. 中文查询召回外文资料的排序、术语处理、引用回溯和质量门槛不够可测。
4. 下载只描述按钮跳转，缺少版本清单、不可用状态、完整性校验、发布回滚和深链规则。
5. 当前官网是 `app/page.tsx` 中的单页客户端原型；语言为组件状态，尚无 `/zh`、`/en` 可索引路由，下载配置、平台识别、弹窗与事件层也未形成独立模块。
6. 文档中 `v0.1.2`、安装包路径、系统要求及“已支持”能力必须由发布事实确认，不应作为永久硬编码文案。

## 2. 范围与术语

### 2.1 本期范围

- 官网以中英文完整解释多语言资料导入、识别、中文理解层、中文索引、中文检索和原文举证。
- 官网根据设备推荐 macOS Apple Silicon 或 Windows x64 安装包，并允许用户查看全部版本。
- 能力、语言、模型和安装包状态由集中配置驱动，支持 `available`、`beta`、`experimental`、`planned`、`unavailable`。
- 建立可执行的内容、功能、质量、安全、SEO、埋点与发布验收标准。

### 2.2 非范围

- 官网不负责执行资料翻译、索引或检索，仅展示已验证的产品能力。
- 不承诺任意语言、任意格式和任意设备均可用。
- 不通过 User-Agent 判断 Mac 芯片架构；只能推荐当前唯一的 Apple Silicon 包并明确限制。
- 不在浏览器中校验大文件 SHA-256；官网负责展示官方校验值和校验说明。

### 2.3 关键术语

| 术语 | 定义 |
| --- | --- |
| 原文层 | 导入内容及其来源、页码、章节、时间戳等证据定位信息，不被翻译覆盖。 |
| 中文理解层 | 由系统生成或人工审校的中文标题、摘要、关键词、实体、术语映射及可选译文。 |
| 中文索引 | 面向中文查询建立的全文、关键词、实体、术语及中文语义检索入口。 |
| 跨语言索引 | 使中文查询与外文原文在共享语义空间或查询扩展后匹配的索引。 |
| 自适应处理 | 根据文档/分块语言、内容类型、工作区策略和资料价值选择 L1/L2/L3 路径。 |
| 原文举证 | 搜索或回答能够返回原始文档及可用的页码、章节、段落或时间戳。 |

## 3. 多语言资料库自适应处理需求

### 3.1 输入识别

系统能力声明只有满足下列行为时才可标记为 `available`：

1. 在文档级和分块级分别记录主语言与置信度；混合语言内容记录语言占比或语言片段。
2. 代码、公式、URL、专有名词、引用和表格单元格不得仅因字符分布而误判为文档主语言。
3. 低置信度、无文本、OCR 失败或不支持语言进入明确的降级状态，不自动宣称处理成功。
4. 支持的首期能力表必须逐语言、逐输入类型维护，不使用笼统的“支持多语言”。

建议状态：

```ts
type LanguageProcessingStatus =
  | "pending"
  | "detecting"
  | "processing"
  | "partially_ready"
  | "ready"
  | "needs_review"
  | "failed"
  | "unsupported";
```

### 3.2 策略选择与优先级

策略优先级固定为：`文档覆盖 > 专题覆盖 > 工作区默认 > 系统默认`。用户变更上级默认值时，不应静默覆盖已有文档的显式策略。

| 路径 | 触发条件 | 最小产物 | 降级行为 |
| --- | --- | --- | --- |
| 中文直通 | 主语言为中文且置信度达标 | 规范化正文、中文全文/语义索引 | 保留原文并标记待处理 |
| L1 快速理解 | 外文或混合内容，默认批量入库 | 中文标题、摘要、关键词、实体、原文索引 | 任一派生字段失败不阻塞原文可检索 |
| L2 按需阅读 | 搜索、问答或用户展开命中分块 | 相关分块中文译文及缓存 | 展示原文与“暂无法生成译文” |
| L3 深度中文化 | 用户显式选择或高价值策略触发 | 完整中文内容层、术语一致性处理 | 已完成分块可用，失败分块可重试 |
| 原样保留 | 不支持、低置信度或特殊内容 | 原文、来源、基础元数据 | 提示能力限制，不伪造中文内容 |

### 3.3 数据分层与可追溯性

- 原文、机器译文、AI 摘要、人工审校内容必须使用独立字段或实体，禁止覆盖。
- 每个派生内容记录来源分块、生成时间、模型/策略版本、语言、审校状态和内容哈希。
- 原文更新时，相关派生内容与索引必须标记过期并增量重建。
- 删除文档时，原文、派生内容、向量、缓存、索引及引用映射按同一删除语义清理。
- UI 必须展示内容类型标签；回答引用默认指向原文，译文仅作为阅读辅助。

### 3.4 异常、重试与成本控制

- 单个分块失败不应使整篇文档不可用；页面可呈现“部分就绪”。
- 重试应幂等，不能产生重复译文、重复向量或重复计费。
- L2 缓存键至少包含原文哈希、目标语言、术语表版本和处理策略版本。
- 云模型启用前明确披露将发送的字段范围；关闭云处理后不得继续自动外发。
- 官网仅在本地模型完成相同验收集验证后使用“本地完成识别/索引/翻译”的确定性表述。

## 4. 中文索引与跨语言检索需求

### 4.1 索引组成

中文查询至少可走以下通道，并由融合排序合并去重：

1. 中文原文全文索引。
2. 中文标题、摘要、关键词、实体和标签索引。
3. 术语表驱动的中文查询扩展与别名索引。
4. 中文语义向量索引。
5. 外文原文跨语言语义索引。

每条召回结果应携带 `matchedField`、`matchedLanguage`、`matchChannel`、`sourceChunkId` 和可解释的命中片段。官网演示可以简化分数，但不得虚构不存在的检索通道。

### 4.2 中文分词与规范化

- 统一处理简繁体、全半角、大小写、Unicode 规范化、数字与单位表达。
- 中文索引不得只按单字符切分；分词器或 n-gram 策略需通过中文检索评测。
- 中英文混合术语（如 `端侧 AI`、产品名、缩写）应保留原形并建立别名映射。
- 术语表支持工作区级覆盖，更新后可触发受影响索引的增量重建。
- 标题、正文、摘要、关键词、实体的权重可配置，避免 AI 摘要压过原文证据。

### 4.3 排序与去重

- 同一原文分块由多个通道命中时合并为一条结果，保留通道解释。
- 默认排序综合关键词、语义、术语、时间、价值及来源质量；权重必须版本化。
- 机器译文和 AI 摘要不得成为唯一引用依据；引用必须能回到原始分块。
- 查询无结果时允许展示拼写、术语或语言建议，不得生成伪搜索结果。

### 4.4 质量验收门槛

上线前由产品与算法冻结中文、英文、混合、日文/韩文实验集，至少覆盖短查询、术语查询、长问题、专有名词和否定表达。

| 指标 | 发布要求 |
| --- | --- |
| Recall@10 / nDCG@10 | 由算法负责人填写基线与门槛；未填写不得宣称 `available` |
| 原文引用可达率 | P0 样本 100% 可回到原始文档；定位信息按格式能力展示 |
| 索引一致性 | 新增、更新、删除后均不出现旧译文或孤儿索引 |
| 混合语言召回 | 中文查询可同时出现相关中文与外文结果，无重复分块 |
| 术语一致性 | 指定术语表的关键术语在标题、摘要、译文和查询扩展中一致 |

日文、韩文、繁体中文未分别通过质量门槛时，官网统一标为 `Experimental`，不可与简体中文/英文并列为正式支持。

## 5. 官网功能与内容补充

### 5.1 能力状态配置

新增单一配置源，营销组件不得硬编码“已支持”：

```ts
type CapabilityState = "available" | "beta" | "experimental" | "planned" | "unavailable";

interface CapabilityConfig {
  multilingualDetection: CapabilityState;
  chineseIndex: CapabilityState;
  crossLanguageRetrieval: CapabilityState;
  originalEvidence: CapabilityState;
  localProcessing: CapabilityState;
  languages: Record<string, CapabilityState>;
  lastVerifiedAt: string;
}
```

规则：`planned`/`unavailable` 不进入主按钮利益点；`beta`/`experimental` 必须紧邻显示状态，不得仅在 FAQ 脚注说明。

### 5.2 Locale 与 SEO

- P0 使用 `/zh` 和 `/en` 独立路由；根路径只做基于偏好的可覆盖推荐，不以 IP 强制跳转。
- 切换语言时保持等价路径、查询参数和锚点，例如 `/zh#multilingual` ↔ `/en#multilingual`。
- 服务端输出对应语言的正文、`lang`、Title、Description、canonical 和双向 `hreflang`；另提供 `x-default`。
- 用户显式选择优先于浏览器语言，并可持久化；无 JavaScript 时语言页仍可访问和索引。
- 中英文 FAQ JSON-LD 分别与可见文案完全一致。

### 5.3 无障碍与内容真实性

- 下载弹窗使用正确的 dialog 语义，打开后聚焦首个可操作项，关闭后归还焦点，支持 `Esc`。
- 平台图标设为装饰，按钮文本同时表达平台和架构；不能只靠颜色区分原文/译文。
- “查看原文”可用键盘操作，并显示来源语言与内容类型。
- 版本、发布日期、大小、系统要求、签名/公证状态只有在发布清单存在时展示。

## 6. 下载功能需求

### 6.1 发布清单为唯一事实源

下载 UI 从版本清单读取，不在 Navbar、Hero、CTA 分别硬编码版本。建议结构：

```ts
interface DesktopRelease {
  version: string;
  channel: "stable" | "beta";
  publishedAt: string;
  platform: "macos" | "windows";
  architecture: "arm64" | "x64";
  fileName: string;
  url: string;
  sizeBytes: number;
  sha256: string;
  minimumOs: string;
  signed: boolean;
  notarized?: boolean;
  status: "available" | "coming_soon" | "withdrawn";
}
```

若清单缺失、字段未确认、URL 健康检查失败或版本被撤回，主 CTA 降级为“查看下载状态/查看全部版本”；存在其他可用安装包时引导用户下载，完全无可用版本时显示“下载暂不可用”，不得继续指向旧包或 404。

### 6.2 平台推荐规则

1. 仅在客户端检测平台，SSR 首屏展示中性“下载客户端”。
2. macOS 推荐 ARM64 DMG，同时明确“仅 Apple Silicon”；不得声称已识别芯片。
3. Windows 推荐 x64 EXE。
4. Linux、移动设备、爬虫和未知平台打开全部版本弹窗，不自动开始下载。
5. 自动检测只影响推荐顺序，不隐藏其他平台，不作为权限或兼容性判断。
6. 用户手动选择后本次会话保持选择，但不得覆盖系统兼容性警告。

### 6.3 下载行为与失败处理

- 具体安装包链接必须是普通可复制的 HTTPS `href`，支持新标签页和无 JavaScript 访问。
- 跨域 CDN 不依赖 HTML `download` 属性，服务端返回正确的 `Content-Type`、`Content-Length`、`Content-Disposition`、缓存头与范围请求支持。
- 点击后记录意图事件；是否真正下载成功不能仅靠前端点击推断，成功量由 CDN/对象存储日志统计。
- 404/5xx、清单超时或离线时显示可重试错误及全部版本入口，不循环自动下载。
- 文件名版本化；新版本发布不覆盖旧对象。撤回版本后更新清单并保留审计记录。
- 展示 SHA-256、签名/公证状态和校验说明；DMG 必须签名及 notarization，EXE 必须代码签名后才可标为稳定版。

### 6.4 下载分析事件

事件去除潜在个人信息，不上报完整 User-Agent、IP、下载 URL 查询参数或本地路径。

| 事件 | 触发 | 必填属性 |
| --- | --- | --- |
| `download_recommendation_shown` | 平台识别完成且推荐可见 | `locale`, `placement`, `detected_platform`, `release_id` |
| `download_dialog_opened` | 用户打开全部版本 | `locale`, `placement`, `detected_platform` |
| `download_started` | 用户点击具体安装包 | `locale`, `placement`, `platform`, `architecture`, `version`, `channel` |
| `download_unavailable_seen` | 推荐包不可用 | `locale`, `platform`, `reason` |
| `checksum_copied` | 用户复制校验值 | `platform`, `version` |

同一次展示只记录一次；前端 `download_started` 代表点击意图，指标命名和仪表盘不得解释为安装成功。

## 7. 验收场景（Given / When / Then）

### 7.1 多语言展示

- Given 能力状态为 `beta`，When 用户进入多语言模块，Then 特性旁明确显示 Beta，FAQ 与主文案口径一致。
- Given 日文状态为 `experimental`，When 用户查看支持语言，Then 不得显示为正式支持或与中英文共用“已支持”标签。
- Given 用户从 `/zh#multilingual` 切换 English，Then 到达 `/en#multilingual` 且焦点/滚动位置可理解。

### 7.2 中文索引演示

- Given 中文查询演示命中英文资料，When 展开结果，Then 同时显示中文理解内容、`英文原文`、`跨语言召回`和可操作的“查看原文”。
- Given 派生中文内容不可用，When 展示结果，Then 仍展示原文并明确提示，不用模拟译文填充。
- Given 同一分块由多通道召回，When 展示列表，Then 只出现一条结果并可查看命中通道。

### 7.3 下载

- Given macOS 浏览器，When hydration 完成，Then 推荐 Apple Silicon DMG，且按钮和弹窗均显示架构限制。
- Given Intel Mac 用户，When 打开全部版本，Then 看到当前无兼容包提示及其他平台下载入口，不被引导安装 ARM64 包。
- Given Windows x64 浏览器，When 点击推荐下载，Then 链接指向清单中的可用稳定版并记录一次点击意图。
- Given Linux、iOS、Android 或未知平台，When 点击主 CTA，Then 打开版本选择，不自动下载。
- Given 发布清单中的版本为 `withdrawn` 或 URL 检查失败，When 页面加载，Then 不提供该包下载，显示状态与替代入口。
- Given JavaScript 禁用，When 访问独立下载页，Then 仍能看到并访问所有 `available` 安装包链接。

## 8. 与当前官网代码的差距

当前实现的主要差距如下，开发排期应以此为基线：

| 项目 | 当前状态 | 目标 |
| --- | --- | --- |
| 页面结构 | 单个 `app/page.tsx` 承载文案、状态和组件 | 拆出内容资源、能力配置、多语言模块及下载组件 |
| 语言切换 | `zh/en` 本地 React 状态 | `/zh`、`/en` 服务端可索引路由并保持锚点 |
| 多语言模块 | 未形成独立模块 | `#multilingual` 完整机制、策略、证据演示 |
| 下载 | 无集中发布清单和平台推荐链路 | 配置驱动按钮、全部版本弹窗、失败降级、事件 |
| SEO | 无按 locale 独立元数据证据 | canonical、hreflang、FAQ JSON-LD、服务端正文 |
| 测试 | 未见对应测试 | 平台检测单测、组件交互、路由/SEO、下载链接冒烟测试 |

## 9. 实施优先级

### P0（上线阻断）

1. 产品/算法填写语言能力矩阵和中文检索质量门槛，确认哪些文案可标 `available`。
2. 建立 `/zh`、`/en` 路由、类型化文案和能力状态配置。
3. 实现完整多语言模块、中文索引与原文举证演示，未完成能力显示状态。
4. 建立下载发布清单、平台推荐、全部版本弹窗、失败降级及独立下载页。
5. 确认安装包版本、最低系统、大小、SHA-256、签名、公证和生产 URL。
6. 完成 SEO、无障碍、事件去重及本稿 Given/When/Then 验收。

### P1（上线后增强）

1. 兼容性说明页和版本历史。
2. 脱敏真实检索演示与质量说明。
3. 下载页增加平台、架构、系统要求、版本、文件大小与校验信息，帮助用户自主选择安装包。
4. CDN 下载日志与前端点击意图漏斗对账。

5. L1/L2/L3 是否已在产品中可配置，还是仅为官网解释模型。
6. 是否已有分析平台与统一 `track` 接口；没有时不得在组件中散落临时实现。
7. 生产下载来源采用 Release、CDN 还是对象存储，以及撤回和回滚负责人。

## 11. 完成定义

本需求只有同时满足以下条件才算完成：

- 官网展示的每项多语言与中文索引声明均可映射到能力状态和验收证据。
- 用户能理解“原文层—中文理解层—中文/跨语言索引—中文使用—原文举证”的完整闭环。
- 中英文页面可独立访问、索引、分享并保持等价锚点。
- 支持设备得到正确且不误导的安装包推荐；不支持设备得到安全降级路径。
- 安装包元数据、完整性、安全签名、可用性和撤回机制均通过发布检查。
- 自动化测试覆盖平台检测、能力状态、locale 路由、下载失败、弹窗无障碍和核心 SEO。
