# 阶段四：标签、实体、分块与向量化详细开发文档

版本：V0.1  
所属项目：双模式 AI 知识资产引擎  
阶段定位：将已经解析、清洗、摘要后的 `document` 转化为可检索、可聚合、可问答、可被 Agent 调用的知识单元。  
前置阶段：阶段三「文档解析、清洗与 AI 摘要」  
后续阶段：阶段五「混合检索与 RAG 问答」

---

## 1. 阶段四总目标

阶段三已经完成了从 `source` 到 `document` 的转换，系统已经拥有较干净的正文、Markdown、摘要、一句话结论、关键要点和基础评分。

阶段四的目标是进一步把 `document` 加工成可计算、可检索、可聚类、可问答的知识结构。

一句话概括：

> 阶段四不是继续“读文档”，而是把文档拆成知识单元，并为每个知识单元建立标签、实体、分块和向量索引。

本阶段完成后，系统应具备：

1. 自动推荐标签；
2. 支持用户确认、合并、删除和新增标签；
3. 从文档中抽取轻量实体；
4. 建立文档与实体之间的关系；
5. 按标题、段落、语义边界进行文档分块；
6. 为每个 chunk 生成 embedding；
7. 在本地和云端分别建立向量索引；
8. 支持索引重建、模型切换、失败重试；
9. 为阶段五混合检索和 RAG 问答提供可靠基础。

---

## 2. 阶段四在整体系统中的位置

整体链路如下：

```text
Inbox
  ↓
Source
  ↓
Parse / Clean / Normalize
  ↓
Document
  ↓
阶段四：标签、实体、分块与向量化
  ↓
Document Tags / Entities / Chunks / Embeddings / Vector Index
  ↓
阶段五：混合检索与 RAG 问答
```

阶段四是知识引擎从“资料管理系统”升级为“知识检索系统”的关键步骤。

如果阶段四设计不稳，阶段五会出现：

1. 搜索结果不准；
2. RAG 引用混乱；
3. 长文无法准确定位；
4. 相似资料难以聚合；
5. 标签体系污染；
6. 向量索引难以重建；
7. 后续 Agent 调用知识库时缺乏可控上下文。

## 3. 阶段四开发范围

### 3.1 本阶段要做

本阶段应完成以下模块：

1. 标签模型与标签推荐；
2. 文档标签关联；
3. 实体抽取；
4. 文档实体关联；
5. 标题感知分块；
6. chunk 元数据生成；
7. embedding 生成；
8. 本地向量索引；
9. 云端 pgvector 索引；
10. embedding 任务队列；
11. 索引状态管理；
12. 索引重建机制；
13. 标签、实体、分块、向量化后台任务；
14. 前端状态展示与人工修正入口。

### 3.2 本阶段不做

以下内容暂不作为阶段四重点：

1. 不做复杂知识图谱编辑器；
2. 不做完整实体消歧系统；
3. 不做大型 ontology 本体系统；
4. 不做跨工作区实体合并；
5. 不做复杂实体关系推理；
6. 不做高级 GraphRAG；
7. 不做多模态图片 embedding 的完整检索；
8. 不做音频语义分段；
9. 不做团队级标签权限管理；
10. 不做复杂向量数据库运维平台。

### 3.3 MVP 边界

MVP 只需要保证：

```text
Document
  → 推荐标签
  → 抽取轻量实体
  → 生成 chunks
  → 生成 embeddings
  → 建立可查询向量索引
```

不追求实体抽取 100% 准确，也不追求标签体系一开始就完美。

## 4. 核心设计原则

### 4.1 先稳后准

阶段四首先要保证数据结构稳定、状态可追踪、索引可重建。

标签和实体可以逐步优化，但底层模型不能频繁推倒重来。

### 4.2 标签是用户可见资产，embedding 是系统底层资产

标签用于用户浏览、筛选和组织；embedding 用于系统检索和问答。

两者都重要，但职责不同：

```text
标签：解释性强，适合用户理解和筛选。
向量：召回能力强，适合语义检索和 RAG。
实体：适合聚合、关联、过滤和后续图谱。
```

### 4.3 分块必须保留上下文

chunk 不能只是机械切割文本。

每个 chunk 必须带有：

1. 所属文档；
2. 所属 source；
3. 所属 topic；
4. 标题路径；
5. chunk 序号；
6. 原文偏移；
7. 前后文关联；
8. 用于引用的定位信息。

### 4.4 embedding 必须可版本化

embedding 与模型强绑定。

如果模型更换，旧 embedding 不能默默复用。

必须记录：

1. embedding_model；
2. embedding_dimension；
3. embedding_provider；
4. embedding_version；
5. chunk_text_hash；
6. generated_at。

### 4.5 本地与云端使用同一领域模型

本地 SQLite 和云端 PostgreSQL 的字段应尽量一致。

差异只体现在：

1. 向量字段存储方式；
2. 索引引擎；
3. 队列实现；
4. 文件存储路径；
5. Worker 部署方式。

## 5. 阶段四整体流程

```text
document.ai_status = done
  ↓
TagWorker
  ↓
推荐标签 / 标签归一 / document_tag 入库
  ↓
EntityWorker
  ↓
实体抽取 / 实体归一 / document_entity 入库
  ↓
ChunkWorker
  ↓
标题感知分块 / chunk 元数据生成
  ↓
EmbeddingWorker
  ↓
批量生成 embedding
  ↓
VectorIndexWriter
  ↓
写入本地或云端向量索引
  ↓
document.index_status = done
```

建议拆分为 4 个独立任务，而不是一个大任务：

```text
TAGGING_JOB
ENTITY_EXTRACTION_JOB
CHUNKING_JOB
EMBEDDING_JOB
```

这样做的好处：

1. 任一环节失败不影响前面结果；
2. 可以单独重跑；
3. 方便调试；
4. 方便后续升级 prompt 或模型；
5. 便于排查性能瓶颈。

## 7. 标签模块设计

## 7.1 标签模块目标

标签模块用于帮助用户快速理解资料主题、过滤资料、组织专题和发现知识聚类。

标签来源包括：

1. AI 推荐；
2. 规则生成；
3. 用户手动添加；
4. 从导入来源继承；
5. 从 topic 自动继承；
6. 后续从实体反推。

标签不应只是字符串，而应成为长期可维护的知识组织资产。

---


MVP 可先简化为：

```text
topic

```

## 7.6 AI 标签推荐 Prompt

### 输入

```json
{
  "title": "文档标题",
  "summary": "中文摘要",
  "one_sentence_conclusion": "一句话结论",
  "key_points": ["要点1", "要点2"],
  "content_preview": "正文前 3000 字"
}
```

### 输出

```json
{
  "tags": [
    {
      "name": "AI Agent",
      "type": "technology",
      "confidence": 0.92,
      "reason": "文章核心讨论 AI Agent 的产品化和企业应用。"
    },
    {
      "name": "企业服务",
      "type": "industry",
      "confidence": 0.81,
      "reason": "文章多次涉及企业级落地场景。"
    }
  ]
}
```

### 标签推荐要求

1. 每篇文档推荐 3-8 个标签；
2. 不要推荐过宽泛标签，如“科技”“互联网”；
3. 不要推荐与内容无关的热门标签；
4. 标签应优先复用已有标签；
5. 标签应支持中英文混合，但 normalized_name 必须统一；
6. 置信度低于 0.5 的标签不自动挂载。

## 8.8 实体误差控制

AI 抽取实体容易产生以下问题：

1. 把普通词当实体；
2. 把标题短语当公司名；
3. 中文翻译名称与英文名称重复；
4. 同一个产品被抽成多个实体；
5. 对小众实体产生幻觉。

控制策略：

1. 只保留 importance >= 0.4 的实体；
2. 只保留 confidence >= 0.6 的实体；
3. 对短词和泛词设置黑名单；
4. 允许用户手动合并；
5. 保留 mention_examples 方便审核；
6. 实体详情页显示“来源文档”。

## 9. 文档分块模块设计

## 9.1 分块模块目标

分块是 RAG 效果的基础。

分块目标不是把文本均匀切开，而是在尽量保留语义完整性的前提下，生成适合检索和引用的 chunk。

好的 chunk 应该具备：

1. 主题相对集中；
2. 长度适中；
3. 包含必要上下文；
4. 可独立被检索；
5. 可在回答中引用；
6. 能回溯到原文位置。

---

## 9.4 分块策略

推荐使用标题感知分块：

```text
Markdown
  ↓
解析标题层级
  ↓
按 section 切分
  ↓
段落合并
  ↓
超过 chunk_size 时二次切分
  ↓
增加 overlap
  ↓
生成 chunk metadata
```

推荐参数：

```text
chunk_size = 800 tokens
chunk_overlap = 120 tokens
min_chunk_size = 150 tokens
max_chunk_size = 1200 tokens
```

## 9.5 分块算法伪代码

```ts
function chunkDocument(document: Document): DocumentChunk[] {
  const markdown = document.content_markdown ?? document.content_text
  const sections = parseMarkdownSections(markdown)
  const chunks: DocumentChunk[] = []

  for (const section of sections) {
    const paragraphs = splitIntoParagraphs(section.content)
    let buffer = []
    let tokenCount = 0

    for (const paragraph of paragraphs) {
      const pTokens = countTokens(paragraph)

      if (tokenCount + pTokens <= CHUNK_SIZE) {
        buffer.push(paragraph)
        tokenCount += pTokens
      } else {
        chunks.push(createChunk(buffer, section.headingPath))
        buffer = createOverlapBuffer(buffer, CHUNK_OVERLAP)
        buffer.push(paragraph)
        tokenCount = countTokens(buffer.join('\n\n'))
      }
    }

    if (buffer.length > 0) {
      chunks.push(createChunk(buffer, section.headingPath))
    }
  }

  return linkPrevNext(normalizeChunks(chunks))
}
```

## 9.6 Markdown 标题路径

对于以下 Markdown：

```markdown
# AI Agent 产品化趋势

## 企业应用

### 客服场景

正文内容...
```

chunk 的 `heading_path` 应为：

```text
AI Agent 产品化趋势 > 企业应用 > 客服场景
```

chunk_title 可取最末级标题：

```text
客服场景
```

这样在 RAG 引用中可以展示：

```text
来源：《AI Agent 产品化趋势》/ 企业应用 / 客服场景
```

## 9.7 特殊内容处理

### 9.7.1 表格

表格不要简单拆散。

处理策略：

1. 小表格整体保留；
2. 大表格按行分组；
3. 表头必须随每个表格 chunk 保留；
4. 表格 chunk 的 metadata 标记为 `content_type = table`。

### 9.7.2 代码块

代码块尽量整体保留。

如果代码块过长：

1. 按函数或类切分；
2. 不在语法结构中间切断；
3. metadata 标记语言。

### 9.7.3 列表

长列表可按条目分组。

但不要把列表标题与列表项拆开。

### 9.7.4 PDF 页码

PDF 来源应尽量记录页码。

字段：

```text
page_start
page_end
```

如果解析器暂时不能提供页码，可以留空。

### 9.7.5 图片说明

图片 OCR 与视觉理解后置。

MVP 可只保留：

```text
![image](file_id)
图片说明：xxx
```

如果图片没有说明，暂不生成独立 chunk。

## 9.8 分块质量评估

每次 chunking 后计算基础指标：

```text
chunk_count
avg_token_count
max_token_count
min_token_count
too_short_chunk_count
too_long_chunk_count
empty_chunk_count
```

建议写入 `documents.chunk_status_detail` 或 `job.result_json`。

质量判断：

| 指标 | 风险 |
|---|---|
| chunk_count = 0 | 分块失败 |
| avg_token_count < 100 | 过度切分 |
| max_token_count > 1500 | 可能超过检索上下文预算 |
| too_short_chunk_count 多 | 噪音多 |
| heading_path 为空过多 | Markdown 结构差 |

## 10. Embedding 模块设计

## 10.1 Embedding 模块目标

Embedding 用于把 chunk 转为向量，支持语义检索。

阶段四不直接做问答，但必须为阶段五提供：

1. 可检索 chunk；
2. 向量相似度；
3. 元数据过滤；
4. 索引状态；
5. 可重建能力。

---

## 10.2 embedding 存储方案

### 方案 A：embedding 直接存 document_chunks

优点：

1. 实现简单；
2. 表少；
3. MVP 快速。

缺点：

1. 不支持多模型并存；
2. 模型切换时难管理；
3. 字段较重。

### 方案 B：独立 chunk_embeddings 表

推荐方案。

优点：

1. 支持多 embedding 模型；
2. 支持重建索引；
3. 支持 A/B 测试；
4. 支持本地与云端差异存储。

---

## 10.3 chunk_embeddings 表设计

```sql
CREATE TABLE chunk_embeddings (
    id                    TEXT PRIMARY KEY,
    workspace_id          TEXT NOT NULL,
    document_id           TEXT NOT NULL,
    chunk_id              TEXT NOT NULL,

    provider              TEXT NOT NULL,
```

## 10.6 Embedding 文本构造

不要只把 chunk.content 直接送入 embedding。


## 10.7 Embedding 状态机

chunk embedding 状态：

```text
pending
queued
embedding
embedded
indexing
indexed
failed
stale
```

含义：

| 状态 | 说明 |
|---|---|
| pending | 等待生成 |
| queued | 已加入队列 |
| embedding | 正在生成向量 |
| embedded | 向量已生成 |
| indexing | 正在写入向量索引 |
| indexed | 已完成索引 |
| failed | 失败 |
| stale | chunk 内容或模型变化，需要重算 |

## 10.8 stale 判断

以下情况需要标记为 stale：

1. chunk.content_hash 变化；
2. embedding_model 变化；
3. embedding_dimension 变化；
4. embedding_provider 变化；
5. vector index schema 变化；
6. 用户主动点击重建索引。

判断逻辑：

```ts
if (existing.chunkContentHash !== chunk.contentHash) {
  markEmbeddingStale(chunk.id)
}

if (existing.model !== currentEmbeddingModel) {
  markEmbeddingStale(chunk.id)
}
```

## 11. 向量索引设计

## 11.1 本地向量索引方案

本地模式可选方案：

| 方案 | 优点 | 缺点 | 建议 |
|---|---|---|---|
| SQLite BLOB + 内存相似度 | 实现简单 | 数据大后慢 | 仅 Demo |
| sqlite-vec | SQLite 生态内 | 需集成扩展 | 推荐 MVP |
| sqlite-vss | 已有实践 | 扩展兼容需验证 | 可选 |
| LanceDB | 检索能力强 | 多一个依赖 | 推荐增强版 |
| hnswlib | 性能好 | 需要本地索引文件 | 可选 |

推荐路线：

```text
MVP：SQLite + sqlite-vec
增强：LanceDB
高级：可插拔 VectorStore Adapter
```

## 11.2 云端向量索引方案

云端推荐：

```text

```

原因：

1. 与业务数据库统一；
2. 部署简单；
3. 支持 metadata join；
4. 适合中小规模团队知识库；
5. 后续可迁移到专用向量数据库。

## 16.3 本地模型降级策略

如果本地 embedding 模型不可用：

```text

  ↓
不可用
  ↓
提示用户配置 embedding 模型
  ↓
允许仅做关键词搜索
  ↓
embedding_status = skipped 或 failed
```

不要因为 embedding 不可用导致文档无法导入。

## 17.3 云端成本控制

Embedding 是高频成本项。

控制策略：

1. 按 chunk hash 去重；
2. 批量 embedding；
3. 限制单文档最大 chunk 数；
4. 对低价值文档可延迟向量化；
5. 对 value_score 低于阈值的资料只做关键词索引；
6. 给用户显示处理额度，后续。

## 20. 索引重建机制

## 20.1 为什么需要重建索引

以下情况需要重建：

1. embedding 模型切换；
2. 分块策略变化；
3. chunk 内容变化；
4. 向量库 schema 变化；
5. 本地索引文件损坏；
6. 用户导入大量旧资料；
7. 版本升级。

---

## 20.2 重建范围

```text
单个 document
单个 topic
整个 workspace
指定时间范围
只重建 failed/stale
```

API 参数：

```json
{
  "scope": "workspace",
  "target_id": "ws_xxx",
  "mode": "stale_only",
  "embedding_model": "default"
}
```

---

## 20.3 重建流程

```text
创建 INDEX_REBUILD_JOB
  ↓
扫描目标范围 chunks
  ↓
判断 embedding 是否 stale
  ↓
删除旧 vector index 记录
  ↓
重新 embedding
  ↓
重新 upsert vector index
  ↓
更新 index_state
```

---

## 20.4 vector_index_state 表

```sql
CREATE TABLE vector_index_state (

    total_chunks          INTEGER NOT NULL DEFAULT 0,
    indexed_chunks        INTEGER NOT NULL DEFAULT 0,
    failed_chunks         INTEGER NOT NULL DEFAULT 0,
    stale_chunks          INTEGER NOT NULL DEFAULT 0,

    status                TEXT NOT NULL DEFAULT 'pending',
    last_rebuilt_at       TEXT,
    schema_version        TEXT NOT NULL DEFAULT 'v1',

    created_at            TEXT NOT NULL,
    updated_at            TEXT NOT NULL
);
```

## 21. 性能与容量预估

## 21.1 单文档估算

假设一篇文章：

```text
正文 5000 中文字
约 2500-3500 tokens
chunk_size = 800 tokens
overlap = 120 tokens
```

大约生成：

```text
4-6 个 chunks
```

如果 2000 篇文章：

```text
约 8000-12000 个 chunks
```

这是 SQLite + 本地向量索引可以承受的规模。

---

## 21.2 本地知识库规模建议

| 规模 | 文档数 | chunk 数 | 推荐方案 |
|---|---:|---:|---|
| 小型 | < 1,000 | < 8,000 | SQLite + sqlite-vec |
| 中型 | 1,000-20,000 | 8,000-150,000 | SQLite + LanceDB |
| 大型 | > 20,000 | > 150,000 | 独立向量库或云端 pgvector |

---

## 21.3 性能优化点

1. chunk 批量写入；
2. embedding 批量请求；
3. 向量 upsert 批量提交；
4. 使用 content_hash 跳过重复计算；
5. 大文档分阶段处理；
6. UI 不等待 embedding 完成；
7. Worker 可暂停和恢复；
8. 本地索引单独文件，避免主库膨胀。

## 27. 关键风险与应对

## 27.1 标签污染

风险：AI 推荐过多泛标签，导致标签系统失控。

应对：

1. 限制每篇文档推荐数量；
2. 标签归一化；
3. 低置信度不挂载；
4. 用户确认机制；
5. 标签合并工具。

---

## 27.2 实体幻觉

风险：AI 抽取不存在的公司、产品或技术。

应对：

1. 保留 mention_examples；
2. 设置 confidence 阈值；
3. 实体黑名单；
4. 用户可删除；
5. 重要实体可标记 verified。

---

## 27.3 分块质量差

风险：chunk 过短、过长或语义断裂。

应对：

1. 标题感知分块；
2. 段落优先；
3. overlap；
4. 分块质量指标；
5. 分块调试页。

---

## 27.4 Embedding 成本高

风险：大量文档导致云端 embedding 成本上升。

应对：

1. 批量 embedding；
2. content_hash 去重；
3. 低价值资料延迟处理；
4. 本地 embedding 优先；
5. 用户额度限制，后续。

---

## 27.5 向量维度不一致

风险：切换模型后不同维度向量写入同一索引导致错误。

应对：

1. 记录 dimension；
2. 每个模型单独 index namespace；
3. 写入前校验维度；
4. 模型切换触发重建。

---

## 27.6 本地索引损坏

风险：应用异常退出导致索引文件不一致。

应对：

1. 主库保存 chunk 与 embedding 状态；
2. 索引可重建；
3. upsert 使用事务；
4. 提供“修复索引”按钮。

---

## 28. 推荐开发顺序

阶段四推荐开发顺序：

```text
1. tags / document_tags 表
2. document_chunks 表
3. Markdown 分块器
4. chunk 状态与展示
5. EmbeddingProvider
6. chunk_embeddings 表
7. VectorStore Adapter
8. 自动 Job 编排
9. AI 标签推荐
10. 实体抽取
11. 索引重建
12. 前端调试与管理页面
```

原因：

1. 分块和 embedding 是阶段五检索的硬前提；
2. 标签和实体可以逐步增强；
3. 先打通 chunk → embedding → index 闭环；
4. 再提升标签和实体质量。

## 29. MVP 最小闭环

阶段四 MVP 最小闭环如下：

```text

  ↓

  ↓
```

## 30. 阶段四完成后的系统能力

完成阶段四后，系统将从“文档资料库”升级为“可检索知识库”。

具备以下能力：

1. 用户可以按标签筛选资料；
2. 用户可以看到文档中的公司、产品、技术等实体；
3. 系统可以把长文拆成可引用的知识块；
4. 系统可以为每个知识块生成语义向量；
5. 本地和云端都具备语义索引能力；
6. 后续 RAG 问答可以精准召回 chunk；
7. Agent 可以调用知识库时获得更稳定的上下文。

阶段四的核心价值不是“看起来更智能”，而是为后续所有知识复用场景建立基础设施。

## 31. 阶段四一句话总结

> 阶段四要把文档从“可阅读内容”加工成“可计算知识单元”：标签用于组织，实体用于关联，chunk 用于引用，embedding 用于语义检索，向量索引用于支撑 RAG 与 Agent 调用。
