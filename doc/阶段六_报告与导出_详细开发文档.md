# 阶段六：报告与导出详细开发文档

版本：V0.1  
所属项目：双模式 AI 知识资产引擎  
所属阶段：阶段六  
前置阶段：阶段一「双模式底座改造」、阶段二「资料导入与 Inbox」、阶段三「文档解析、清洗与 AI 摘要」、阶段四「标签、实体、分块与向量化」、阶段五「混合检索与 RAG 问答」  
阶段目标：把已经进入知识库的资料，转化为可复用、可追溯、可导出、可进入 Obsidian / Agent / 工作流的结构化报告与知识产物。

---

## 1. 阶段定位

阶段六不是简单做一个“导出按钮”，也不是把若干摘要拼接成一篇文章，而是要把前五个阶段沉淀下来的知识资产转化为用户真正可消费、可交付、可复盘、可迁移的输出物。

前五个阶段解决的是：

```text
资料进来
  ↓
解析清洗
  ↓
摘要结构化
  ↓
分块向量化
  ↓
检索问答
```

阶段六要解决的是：

```text
知识资产如何定期汇总？
知识资产如何形成专题报告？
知识资产如何导出到 Markdown / Obsidian？
报告结论如何追溯来源？
用户如何把知识库中的内容拿出去继续使用？
```

因此，本阶段的核心价值是：

> **让知识库从“能查、能问”升级为“能输出、能沉淀、能迁移”。**

阶段六完成后，系统应具备以下能力：

1. 用户可以基于专题、时间范围、标签、实体或查询条件生成报告；
2. 系统可以生成日报、周报、专题报告；
3. 报告内容必须带来源引用；
4. 报告可以保存到本地 / 云端工作区；
5. 报告可以导出为 Markdown；
6. 报告可以按 Obsidian Vault 结构导出；
7. 报告生成过程可追溯、可重试、可调试；
8. 本地模式和云端模式共用统一 ReportService 与 ExportService；
9. 后续 MCP / Agent 可以读取报告或触发报告生成。

## 2. 本阶段与前后阶段关系

5. RAGService 可以基于上下文生成回答；
6. CitationBuilder 可以输出可追溯引用；
7. FileStorage 可以保存导出文件；

```text
  ↓

  ↓
返回结构化周报
```

## 3. 阶段目标与非目标

### 3.1 阶段目标

| 目标 | 说明 |
|---|---|
| 报告生成 | 支持日报、周报、专题报告 |
| 引用追溯 | 报告结论必须追溯到 document / chunk / source |
| 模板化 | 支持不同报告类型的结构模板 |
| 可导出 | 支持 Markdown、Obsidian、JSON，后续扩展 PDF / Word |
| 双模式兼容 | 本地 SQLite / Vault 与云端 PostgreSQL / S3 共用接口 |
| 异步任务 | 报告生成和导出必须支持后台 Job、进度、失败重试 |
| 可编辑 | 报告生成后允许用户编辑标题、正文、标签、归属专题 |
| 可复用 | 报告可被后续 RAG、MCP、Agent、导出模块复用 |

### 3.2 本阶段不做

阶段六暂不做以下内容：

1. 不做复杂所见即所得富文本编辑器；
2. 不做企业级审批流；
3. 不做多人协同编辑；
4. 不做报告模板市场；
5. 不做复杂排版设计器；
6. 不做自动发布到公众号、知乎、微博等外部平台；
7. 不做严格学术论文引用格式；
8. 不做复杂图表生成，除非数据已经结构化；
9. 不做 Word / PDF 作为 P0，后续可扩展；
10. 不做跨工作区聚合报告，除非用户明确选择。

## 6. 报告类型设计

### 6.1 报告类型枚举

```text
daily
weekly
topic
query
collection
custom
```

### 6.2 日报 Daily Report

日报用于回答：

```text
今天 / 指定日期，某个专题或整个工作区新增了哪些重要信息？
```

适用场景：

1. AI 资讯日报；
2. 行业动态日报；
3. 竞品监控日报；
4. 学习资料日报；
5. 项目资料日报。

输入条件：

```ts
DailyReportInput {
  workspaceId: string
  topicId?: string
  date: string
  sourceTypes?: string[]
  tagIds?: string[]
  minValueScore?: number
  language?: "zh-CN" | "en-US"
}
```

推荐结构：

```markdown
# {日期} {专题名} 日报

## 一句话总结

## 今日最重要的 3-5 条信息

## 关键趋势

## 重要资料摘要

## 风险与不确定性

## 值得后续深挖的问题

## 来源列表
```

### 6.3 周报 Weekly Report

周报用于回答：

```text
过去一周，某个专题有哪些持续变化、趋势和结论？
```

周报不同于日报，不能只是把每天内容拼接，而应强调：

1. 趋势；
2. 变化；
3. 连续事件；
4. 高频实体；
5. 值得行动的判断；
6. 未解决问题。

输入条件：

```ts
WeeklyReportInput {
  workspaceId: string
  topicId?: string
  startDate: string
  endDate: string
  sourceTypes?: string[]
  tagIds?: string[]
  entityIds?: string[]
  minValueScore?: number
  language?: "zh-CN" | "en-US"
}
```

推荐结构：

```markdown
# {起始日期} - {结束日期} {专题名} 周报

## 一句话结论

## 本周核心变化

## 重要趋势

## 关键事件时间线

## 高频公司 / 产品 / 技术 / 人物

## 机会信号

## 风险信号

## 下周需要关注的问题

## 来源列表
```

### 6.4 专题报告 Topic Report

专题报告用于回答：

```text
围绕某个主题，基于知识库已有资料，形成一篇结构化分析报告。
```

适用场景：

1. 某个行业研究；
2. 某个技术路线分析；
3. 某个市场机会判断；
4. 某个竞品研究；
5. 某个项目可行性分析；
6. 某个资料集合的综合报告。

输入条件：

```ts
TopicReportInput {
  workspaceId: string
  topicId?: string
  query: string
  startDate?: string
  endDate?: string
  tagIds?: string[]
  entityIds?: string[]
  sourceTypes?: string[]
  minValueScore?: number
  maxSources?: number
  depth: "brief" | "standard" | "deep"
  language?: "zh-CN" | "en-US"
}
```

推荐结构：

```markdown
# {专题标题}

## 摘要

## 一句话结论

## 背景与问题定义

## 关键事实

## 主要观点

## 证据与来源分析

## 机会

## 风险

## 不确定性

## 建议行动

## 后续研究问题

## 来源列表
```

### 6.5 查询报告 Query Report

查询报告是从用户的自然语言问题出发，自动检索并组织成报告。

示例：

```text
帮我基于知识库，整理一份“2026 年本地 AI Agent 工具链趋势报告”。
```

流程：

```text
用户输入 query
  ↓
QueryUnderstandingService 提取主题、时间、实体、意图
  ↓
SearchEngine 多轮召回资料
  ↓
ReportPlanBuilder 生成大纲
  ↓
ReportGenerator 生成报告
```

查询报告是专题报告的增强形态，P0 可以先使用专题报告实现，P1 再单独优化。

## 7. 报告生成流程

### 7.1 标准流程

```text
用户选择报告类型
  ↓
填写报告参数
  ↓
创建 report_job
  ↓
ReportPlanBuilder 生成报告计划
  ↓
ReportSourceSelector 检索候选资料
  ↓
ReportContextBuilder 构建上下文
  ↓
ReportPromptBuilder 生成提示词
  ↓
ModelProvider 生成报告正文
  ↓
ReportCitationBuilder 绑定引用
  ↓
ReportQualityEvaluator 质量检查
  ↓
reports 入库
  ↓
前端展示报告
```

### 7.2 报告计划 Report Plan

报告计划用于让生成过程可解释、可调试。

```ts
ReportPlan {
  id: string
  reportType: "daily" | "weekly" | "topic" | "query" | "custom"
  title: string
  goal: string
  sections: ReportSectionPlan[]
  filters: ReportSourceFilter
  maxSources: number
  maxChunks: number
  modelConfig: ModelConfigSnapshot
}

ReportSectionPlan {
  key: string
  title: string
  purpose: string
  expectedEvidenceType: "facts" | "trends" | "risks" | "opportunities" | "summary"
  minCitations: number
}
```

### 7.3 来源选择策略

来源选择不能只靠向量 Top K，应结合阶段五的混合检索能力。

```text
候选资料来源
  ├── 时间范围过滤
  ├── 专题过滤
  ├── 标签过滤
  ├── 实体过滤
  ├── 来源类型过滤
  ├── value_score 过滤
  ├── keyword search
  ├── vector search
  └── hybrid ranking
```

默认策略：

```text
日报：按 created_at / published_at + value_score + topic 选择
周报：按时间范围 + 高频实体 + value_score + 多样性选择
专题报告：按 query 语义召回 + 关键词召回 + 元数据过滤
```

建议参数：

```ts
ReportRetrievalConfig {
  maxDocuments: 30
  maxChunks: 80
  perDocumentMaxChunks: 5
  minHybridScore: 0.35
  minValueScore: 50
  diversityEnabled: true
  includeDocumentSummary: true
  includeChunkContent: true
}
```

### 7.4 上下文构建策略

报告上下文应分层构建，而不是把所有 chunk 一次性塞入模型。

```text
第一层：报告目标和结构
第二层：资料清单
第三层：文档级摘要
第四层：关键 chunk 证据
第五层：标签 / 实体 / 时间线元数据
第六层：引用编号
```

上下文结构示例：

```markdown
# 报告任务
生成一份关于「{query}」的专题报告。

# 报告结构
- 摘要
- 一句话结论
- 关键事实
- 机会
- 风险
- 建议行动

# 资料清单
[DOC-1] 标题：...
来源：...
日期：...
摘要：...

# 关键证据
[CIT-1] 来自 DOC-1 / CHUNK-3
内容：...

[CIT-2] 来自 DOC-2 / CHUNK-1
内容：...
```

### 7.5 分段生成策略

对于较长专题报告，建议使用分段生成，避免上下文过长和结构失控。

```text
ReportPlan
  ↓
Section 1 生成
  ↓
Section 2 生成
  ↓
Section 3 生成
  ↓
合并全文
  ↓
统一引用校验
  ↓
统一风格润色
```

P0 可先采用一次性生成：

```text
brief / standard 报告：一次生成
```

P1 再支持：

```text
deep 报告：分段生成
```

## 9. AI 提示词设计

### 9.1 报告生成系统提示词原则

报告生成提示词必须强调：

1. 只能基于给定资料；
2. 不确定内容必须标注；
3. 重要结论必须带引用；
4. 不允许虚构来源；
5. 不允许把推测写成事实；
6. 输出 Markdown；
7. 保留结构化标题；
8. 使用用户指定语言。

### 9.2 通用系统提示词

```text
你是一个严谨的知识资产分析助手。

你必须基于用户提供的资料生成报告。
禁止编造资料、数据、引用、来源、公司名称、人物名称或事实。
如果资料不足，请明确说明“不足以得出结论”。

报告中的关键事实、判断、风险、机会必须尽量绑定引用编号。
引用编号来自上下文中的 [CIT-x]，不得创造不存在的引用编号。

请使用结构清晰的 Markdown 输出。
语言风格应专业、清晰、克制，避免营销化表达。
```

### 9.3 日报提示词

```text
请基于以下资料生成一份日报。

要求：
1. 先给出一句话总结；
2. 提炼今日最重要的 3-5 条信息；
3. 标出关键趋势、风险、不确定性；
4. 不要逐条机械复述资料；
5. 重要结论后附引用编号，例如 [CIT-1]；
6. 如果资料数量较少，请明确说明覆盖范围有限。
```

### 9.4 周报提示词

```text
请基于以下资料生成一份周报。

要求：
1. 重点分析一周内的变化、趋势和连续信号；
2. 不要简单拼接每日摘要；
3. 识别高频实体，包括公司、产品、技术、人物；
4. 输出机会、风险和下周关注点；
5. 重要判断必须带引用；
6. 对资料不足或冲突的地方保持谨慎。
```

### 9.5 专题报告提示词

```text
请基于以下资料生成一份专题报告。

要求：
1. 明确问题定义和分析边界；
2. 总结关键事实；
3. 提炼主要观点；
4. 区分事实、判断和推测；
```

## 14. 报告质量评估

### 14.1 质量评分维度

报告生成后应计算 `quality_score`。

| 维度 | 权重 | 说明 |
|---|---:|---|
| 结构完整度 | 20% | 是否包含模板要求章节 |
| 引用覆盖率 | 25% | 关键结论是否有引用 |
| 来源多样性 | 15% | 是否过度依赖单一文档 |
| 信息密度 | 15% | 是否空泛、重复、套话 |
| 时间范围覆盖 | 10% | 是否覆盖指定时间范围 |
| 风险/不确定性表达 | 10% | 是否区分事实与判断 |
| 格式规范 | 5% | Markdown 是否规范 |

### 14.2 citation_coverage

```text
citation_coverage = 带引用的关键段落数量 / 关键段落总数量
```

关键段落包括：

1. 关键事实；
2. 趋势判断；
3. 风险判断；
4. 机会判断；
5. 建议行动。

### 14.3 低质量报告处理

当质量评分低于阈值：

```text
quality_score < 60
```

处理方式：

1. 标记报告质量较低；
2. 前端提示用户；
3. 允许重新生成；
4. 保存失败原因；
5. 不自动导出，除非用户强制。

常见低质量原因：

1. 召回资料太少；
2. 资料时间范围不匹配；
3. 模型输出格式错误；
4. 引用不足；
5. 报告内容过短；
6. 重复内容过多。

## 20. Job 编排设计

### 20.1 ReportJob Worker

伪代码：

```ts
async function runReportJob(jobId: string) {
  const job = await repo.getReportJob(jobId)

  await repo.updateReportJob(jobId, { status: "planning", progress: 10 })
  const plan = await reportPlanBuilder.build(job.input)

  await repo.updateReportJob(jobId, { status: "retrieving", progress: 25, planJson: plan })
  const sources = await reportSourceSelector.select(plan)

  await repo.updateReportJob(jobId, { status: "building_context", progress: 40 })
  const context = await reportContextBuilder.build(plan, sources)

  await repo.updateReportJob(jobId, { status: "generating", progress: 60 })
  const output = await reportGenerator.generate(plan, context)

  await repo.updateReportJob(jobId, { status: "evaluating", progress: 80 })
  const evaluation = await reportQualityEvaluator.evaluate(output, sources)

  await repo.updateReportJob(jobId, { status: "saving", progress: 90 })
  const report = await reportRepository.saveReport(output, evaluation, sources)

  await repo.updateReportJob(jobId, {
    status: "done",
    progress: 100,
    reportId: report.id,
    finishedAt: now()
  })
}
```

### 20.2 ExportJob Worker

```ts
async function runExportJob(jobId: string) {
  const job = await repo.getExportJob(jobId)

  await repo.updateExportJob(jobId, { status: "preparing", progress: 10 })
  const targets = await exportTargetResolver.resolve(job)

  await repo.updateExportJob(jobId, { status: "rendering", progress: 40 })
  const files = await exporter.render(job.exportType, targets, job.options)

  await repo.updateExportJob(jobId, { status: "packaging", progress: 70 })
  const output = await packager.package(files)

  await repo.updateExportJob(jobId, { status: "saving", progress: 90 })
  const fileObject = await fileStorage.saveFile(output)

  await repo.updateExportJob(jobId, {
    status: "done",
    progress: 100,
    outputFileId: fileObject.id,
    finishedAt: now()
  })
}
```

### 20.3 重试策略

```text
最大重试次数：3
第一次失败：立即重试
第二次失败：延迟 1 分钟
第三次失败：延迟 5 分钟
仍失败：标记 failed，等待用户手动重试
```

不可自动重试的错误：

1. 用户取消；
2. 权限不足；
3. 工作区不存在；
4. 模型配置缺失；
5. 导出目录不可写；
6. 存储空间不足。

## 21. 权限、隐私与安全

### 21.1 本地模式隐私

本地模式默认：

1. 报告只保存在本地；
2. 导出文件只写入本地 Vault；
3. 不上传报告正文；
4. 用户使用云端模型时必须提示内容会发送给模型服务商；
5. 用户可删除报告和导出文件。

### 21.2 云端模式权限

云端模式必须保证：

1. `workspace_id` 强隔离；
2. 用户只能访问自己工作区的报告；
3. 下载 URL 使用短期签名；
4. 导出任务不能跨工作区读取资料；
5. Worker 日志不保存完整原文；
6. 删除工作区时删除报告、引用、导出文件。

### 21.3 敏感报告提示

当报告来源包含以下内容时，应提示用户：

1. 私人笔记；
2. 合同；
3. 财务资料；
4. 身份信息；
5. 客户资料；
6. 未公开项目资料。

提示文案：

```text
该报告可能包含敏感资料。导出或同步前，请确认是否允许将内容写入目标位置。
```

## 23. 验收标准

### 23.1 报告生成验收

1. 用户可以选择专题和时间范围生成日报；
2. 用户可以选择专题和时间范围生成周报；
3. 用户可以输入主题生成专题报告；
4. 报告生成任务有进度状态；
5. 报告生成失败时能看到错误；
6. 报告可以重新生成；
7. 报告正文使用 Markdown；
8. 报告有来源引用；
9. 引用可以定位到 document / chunk；
10. 报告保存到 reports 表。

### 23.2 导出验收

1. 单个报告可以导出 Markdown；
2. 专题可以导出 Obsidian 目录；
3. 导出内容包含 sources.json；
4. 导出文件保存到本地 Vault 或云端对象存储；
5. 导出任务有进度状态；
6. 导出失败可以重试；
7. 本地导出的 Markdown 可直接打开；
8. Obsidian 导出可以被 Obsidian 作为 Vault 打开；
9. JSON 导出格式合法；
10. 导出文件名不包含非法字符。

### 23.3 双模式验收

本地模式：

1. 不登录也能生成报告；
2. 报告保存在 SQLite；
3. Markdown 文件保存在本地 Vault；
4. 不启用云端同步时不上传报告。

云端模式：

1. 报告按 workspace 隔离；
2. 导出文件保存到 S3 / MinIO；
3. 下载链接有权限控制；
4. Worker 可以异步执行报告任务。

混合模式：

1. 本地库可基于手机采集资料生成报告；
2. 报告默认保存在本地；
3. 同步报告到云端必须用户确认。

## 26. 风险与应对

### 26.1 报告变成摘要拼接

风险：系统只是把多篇摘要拼起来，没有综合分析价值。

应对：

1. 报告模板强调趋势、变化、风险、机会；
2. 周报必须做跨文档归纳；
3. 专题报告必须有问题定义和建议行动；
4. Prompt 中禁止机械复述；
5. 引入质量评分检查信息密度。

### 26.2 引用不可靠

风险：模型生成的引用编号和真实来源不一致。

应对：

1. 引用编号由系统预生成；
2. 模型只能使用已有引用编号；
3. 生成后校验引用编号是否存在；
4. 不存在的引用自动移除或标记；
5. 关键段落无引用则降低质量评分。

### 26.3 报告内容幻觉

风险：模型基于常识或外部知识补充了资料中没有的事实。

应对：

1. 系统提示词严格限制来源；
2. 资料不足时要求说明；
3. 低引用覆盖率警告；
4. 保留 retrieval_snapshot；
5. 允许用户查看报告依据。

### 26.4 导出结构混乱

风险：Obsidian 导出目录、文件名、双链混乱，用户难以使用。

应对：

1. P0 先采用简单稳定结构；
2. 每次导出生成 README；
3. 文件名统一安全处理；
4. 保留 export_manifest.json；
5. 用户可选择是否导出 documents。

### 26.5 大报告生成失败

风险：上下文过长导致模型失败或输出中断。

应对：

1. P0 限制 maxDocuments 和 maxChunks；
2. deep 报告采用分段生成；
3. 保存中间结果；
4. 支持重试；
5. 提示用户缩小范围。

### 26.6 本地文件权限问题

风险：导出目录无权限、路径不存在、磁盘空间不足。

应对：

1. 导出前检查目录可写；
2. 检查剩余空间；
3. 错误信息前端可读；
4. 允许用户选择新路径；
5. 不覆盖用户已有文件。

## 27. 推荐最终实现策略

阶段六建议采用以下实现顺序：

```text
先做报告数据模型
  ↓
再做手动报告生成
  ↓
再做引用链和报告详情
  ↓
再做 Markdown 导出
  ↓
再做 Obsidian 导出
  ↓
最后做质量评分、版本、定时生成
```

P0 最小闭环：

```text
选择专题 / 时间范围 / 查询主题
  ↓
调用 SearchEngine 召回资料
  ↓
调用 ModelProvider 生成报告
  ↓
保存 reports
  ↓
展示报告和引用
  ↓
导出 Markdown
  ↓
导出 Obsidian 基础目录
```

本阶段完成后，系统将从“知识问答工具”进一步升级为：

> **能持续沉淀研究成果、生成可交付报告、并保持开放导出能力的个人 / 小团队知识资产平台。**
