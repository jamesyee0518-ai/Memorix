# 双模式 AI 知识资产引擎完整开发文档

版本：V0.1  
定位：本地优先 + 可选云端 + 手机端采集入口的个人 / 小团队 AI 知识资产引擎  
核心路线：路线 1 本地优先为主，路线 3 云端 SaaS 为增强，手机端作为信息源采集入口  

---

## 1. 项目总定位

本项目不再单纯定义为云端知识库，也不再单纯定义为本地桌面工具，而应定义为：

> **双模式 AI 知识资产引擎：支持本地工作区、云端工作区和混合工作区，让用户把网页、PDF、图片、录音、笔记、聊天输入等信息源转化为可检索、可问答、可报告、可被 Agent 调用的个人知识资产。**

核心理念：

```text
本地优先，保护隐私；
云端可选，增强便利；
手机采集，降低信息录入门槛；
统一知识模型，支持未来同步与团队协作。
```

## 2. 产品边界

### 2.1 产品要解决的问题

用户每天从网页、公众号、PDF、聊天、截图、录音、会议、邮件、视频等渠道接收大量信息，但这些信息常常分散在收藏夹、聊天记录、文件夹、笔记软件和大脑记忆中，无法被系统化整理、长期记忆和高效复用。

本项目要解决：

1. 信息源便捷进入系统；
2. 原始资料被自动解析、清洗、摘要和结构化；
3. 用户可以跨资料进行混合检索和 RAG 问答；
4. 系统可以生成日报、周报、专题报告；
5. 知识资产可以导出为 Markdown / Obsidian；
6. 本地 Agent 或云端 Agent 可以调用知识库；
7. 用户可以选择数据在本地、云端或混合保存。

### 2.2 产品不是

MVP 阶段不做以下内容：

1. 不做完整 Notion 替代品；
2. 不做完整 Obsidian 替代品；
3. 不做大型团队协作文档平台；
4. 不做知识交易市场；
5. 不做复杂插件生态；
6. 不做完整移动端知识管理 App；
7. 不做全自动 Agent 执行平台；
8. 不做泛化企业知识中台。

### 2.3 产品应该是

MVP 阶段应聚焦为：

```text
桌面端：主工作台，本地优先，负责资料加工、检索、问答和报告。
Web 端：云端工作区入口，适合轻量用户和多设备访问。
手机端：采集入口，以聊天窗口方式提交信息源。
云端：Inbox、同步、备份、云端处理和团队扩展能力。
MCP / API：让 Agent 可以调用用户授权知识库。
```

## 3. 三种工作区模式

系统必须引入 Workspace 工作区概念。用户不是简单创建一个账号，而是创建一个或多个工作区。

### 3.1 Local Workspace：本地工作区

本地工作区适合重隐私用户、本地模型用户、开发者、研究者。

特点：

1. 数据保存在本机；
2. 使用 SQLite；
3. 文件保存在本地 Vault 文件夹；
4. 支持 Ollama / LM Studio；
5. 支持本地 MCP Server；
6. 不强制注册账号；
7. 可选连接云端 Inbox。

架构：

```text
Tauri Desktop App
  ↓
Local Core Service
  ↓
SQLite
Local Vault
Local Queue
Local Vector Index
  ↓
Ollama / LM Studio / User API Key
  ↓
Local MCP Server
```

### 3.2 Cloud Workspace：云端工作区

云端工作区适合普通用户、多设备用户、团队用户。

特点：

1. 数据保存在云端；
2. 使用 PostgreSQL / pgvector；
3. 文件保存在 S3 / MinIO；
4. 云端 Worker 处理任务；
5. 支持 Web 和手机端访问；
6. 支持账号体系；
7. 后续支持团队协作。

架构：

```text
Web / Mobile App
  ↓
Cloud API Service
  ↓
PostgreSQL + pgvector
Redis / Worker
S3 / MinIO
  ↓
Cloud Model API / User API Key
```

### 3.3 Hybrid Workspace：混合工作区

混合工作区是长期最理想模式。

特点：

1. 本地保存主知识库；
2. 手机端采集先进入云端 Inbox；
3. 桌面端拉取云端 Inbox 写入本地库；
4. 可选同步摘要、索引或报告到云端；
5. 原始敏感资料可选择不上传；
6. 兼顾隐私和便利性。

架构：

```text
Mobile Capture
  ↓
Cloud Inbox
  ↓
Desktop Sync Client
  ↓
Local Workspace
  ↓
Local Processing / RAG / MCP
```

## 11. Inbox 收件箱模块

### 11.1 定位

Inbox 是所有零散输入的缓冲层。

来源包括：

1. 手机聊天窗口；
2. 桌面端手动输入；
3. 浏览器插件，后续；
4. 系统分享，后续；
5. 邮件转发，后续；
6. 文件拖拽。

### 11.2 为什么必须有 Inbox

如果所有采集内容直接进入正式知识库，会导致分类混乱、低质量资料污染索引。

Inbox 负责：

1. 临时保存；
2. 自动识别类型；
3. AI 推荐专题；
4. 用户确认归类；
5. 批量导入；
6. 失败重试。

### 11.3 inbox_items 表

本地 SQLite 和云端 PostgreSQL 均保留此模型。

```sql
CREATE TABLE inbox_items (
    id                  TEXT PRIMARY KEY,
    workspace_id        TEXT NOT NULL,

    file_id             TEXT,

    status              TEXT NOT NULL DEFAULT 'pending',
    suggested_topic_id  TEXT,
    suggested_title     TEXT,
    error_message       TEXT,

    created_from        TEXT,
    created_at          TEXT NOT NULL,
    updated_at          TEXT NOT NULL
);
```

```text

```

### 11.5 status 枚举



## 13. 本地与手机端打通方案

### 13.1 推荐 MVP：云端 Inbox 中转

流程：

```text
手机端采集
  ↓
云端 Inbox
  ↓
桌面端同步拉取
  ↓
写入本地 Workspace
  ↓
本地解析、摘要、检索
```

优点：

1. 体验最好；
2. 手机随时可用；
3. 桌面端离线也可先收集；
4. 技术难度低于局域网直连；
5. 后续容易做多设备同步。

隐私要求：

用户必须明确开启：

```text
允许手机端采集内容上传到云端 Inbox。
```

### 13.2 高级方案：局域网直连

流程：

```text
桌面端启动 Local Server
  ↓
手机扫码配对
  ↓
手机直接发送资料到桌面端
```

适合高级隐私用户，后续实现。

### 13.3 第三方同步目录

支持 WebDAV、iCloud、OneDrive、Syncthing 等，作为高级选项。

## 17. 混合检索与 RAG 问答

### 17.1 检索能力

必须支持：

1. 关键词检索；
2. 向量检索；
3. 标签过滤；
4. 实体过滤；
5. 时间过滤；
6. 来源过滤；
7. 价值评分过滤；
8. 混合排序。

### 17.2 混合打分

```text
final_score = keyword_score * 0.35
            + vector_score * 0.40
            + freshness_score * 0.10
            + value_score_norm * 0.10
            + metadata_score * 0.05
```

### 17.3 RAG 问答流程

```text
用户提问
  ↓
Hybrid Search
  ↓
Top chunks
  ↓
ContextBuilder
  ↓
LLM Answer
  ↓
CitationBuilder
  ↓
返回答案和来源
```

### 17.4 防幻觉原则

1. 无相关资料时拒答；
2. 回答必须基于召回资料；
3. 引用由系统生成；
4. 保留 retrieval_snapshot；
5. 用户可反馈引用错误。

## 25. 关键风险与应对

### 25.1 双模式导致架构复杂

风险：本地和云端两套逻辑割裂。

应对：统一领域模型，业务层依赖 Adapter，不直接依赖具体数据库。

### 25.2 手机端采集隐私争议

风险：本地用户不希望资料经过云端。

应对：云端 Inbox 必须显式开启，后续提供局域网直连。

### 25.3 本地模型体验不稳定

风险：不同用户硬件差异大。

应对：允许本地数据 + 用户自带云端模型 API。

### 25.4 同步复杂度过高

风险：双向同步冲突复杂。

应对：MVP 只做云端 Inbox 到本地单向拉取。

### 25.5 过早做 SaaS

风险：账号、计费、团队、合规拖慢 MVP。

应对：先跑通本地个人闭环，云端只做 Inbox 和可选工作区。

## 27. 产品路线总结

最终产品路线：

```text
路线 1：本地优先，是信任和差异化核心。
路线 3：云端能力，是多设备、手机采集和团队协作增强。
手机端：不是主工作台，而是信息源录入窗口。
Obsidian：不是替代对象，而是导出和开放数据格式。
MCP：是本地 Agent 调用知识库的关键接口。
```

最终产品形态：

```text
桌面端：主工作台，本地优先
Web 端：云端工作区和轻量访问
手机端：聊天式信息采集入口
云端：Inbox、同步、备份、团队扩展
MCP/API：Agent 调用知识库
```

一句话总结：

> **本项目应改造成“本地优先 + 可选云端 + 手机采集”的双模式 AI 知识资产引擎，而不是单纯本地工具，也不是单纯云端 SaaS。**
