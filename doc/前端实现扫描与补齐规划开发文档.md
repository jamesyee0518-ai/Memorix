# 前端实现扫描与补齐规划开发文档

版本：V0.1  
范围：Memorix 本地前端 / Web 工作台  
对照来源：《双模式知识引擎完整开发文档.md》  
当前前端路径：`web/`

---

## 1. 文档目标

本文用于对照“双模式 AI 知识资产引擎”完整规划，梳理当前本地前端的实现状态，并形成可执行的补齐方案。

重点回答三个问题：

1. 当前前端已经实现了哪些规划要求；
2. 哪些能力只是部分实现或占位；
3. 后续如何按优先级补齐本地优先、云端 Inbox、手机采集和混合模式闭环。

---

## 2. 当前前端总体形态

当前前端是一个 Next.js Web 工作台，而不是直接嵌入 Tauri 的桌面前端。

技术栈：

| 模块 | 当前实现 |
|---|---|
| 前端框架 | Next.js 16 |
| UI 框架 | React 19 |
| 语言 | TypeScript |
| 样式 | Tailwind CSS |
| UI 组件 | shadcn / Base UI |
| 图标 | lucide-react |
| 请求库 | axios |
| 状态管理 | Zustand |
| 数据请求 | React Query |
| 后端入口 | `http://localhost:9101/api` |

核心判断：

```text
当前前端 = 本地 API 驱动的 Web 控制台
目标产品 = 本地优先桌面工作台 + 可选云端 + 手机采集入口
```

也就是说，当前前端已经承担了“主工作台”的大部分信息架构和页面能力，但还缺少桌面壳、免登录本地模式、真实云端 Inbox 同步和手机端采集入口。

## 4. 总体差距判断

### 4.1 已经接近完成的部分

```text
Workspace UI
资料导入
桌面 Inbox
文档展示
AI 摘要展示
标签 / 实体 / 分块入口
混合检索
RAG 问答
报告生成
Markdown / Obsidian / JSON 导出
Agent / MCP 管理入口
使用量和反馈系统
```

### 4.2 需要重点补齐的部分

```text
Tauri 桌面壳
本地免登录模式
首次启动完整向导
云端 Inbox 真实同步
手机端聊天式采集
混合模式数据流闭环
全局隐私与数据流向可见性
图片 OCR / 音频转写的前端状态
```

### 4.3 当前产品形态风险

当前前端如果直接交付，会让用户感知为：

```text
一个连接本地后端的 Web 管理台
```

而不是文档目标中的：

```text
本地优先 + 可选云端 + 手机采集的双模式 AI 知识资产引擎
```

因此后续补齐重点不是再增加大量普通管理页面，而是打通“本地可信任”和“手机采集进入本地库”这两个核心体验。

## 5. 补齐路线

## 阶段 A：本地 MVP 体验补齐

目标：让本地模式符合文档验收标准。

### A1. 本地免登录模式

问题：

当前主布局强制检查 token，本地模式无法“不注册账号直接使用”。

方案：

1. 新增本地匿名会话状态；
2. 启动时先请求 `/workspaces/current`；
3. 如果存在本地工作区，允许进入；
4. 如果无工作区，进入 `/setup`；
5. 云端工作区、API Key、云端 Inbox 等能力仍要求登录；
6. 登录状态和本地匿名状态在 UI 上明确区分。

涉及文件：

```text
web/src/app/(main)/layout.tsx
web/src/stores/auth-store.ts
web/src/lib/api.ts
```

验收：

1. 无 token 时可以创建本地工作区；
2. 无 token 时可以进入本地 Dashboard；
3. 访问云端能力时提示登录；
4. 用户能看到当前是“本地匿名模式”。

---

### A2. 首次启动完整向导

问题：

当前创建工作区后直接进入 Dashboard，缺少“创建第一个专题”。

方案：

将 `/setup` 改为分步向导：

```text
Step 1：选择工作区模式
Step 2：配置存储位置
Step 3：配置模型
Step 4：创建第一个专题
Step 5：进入 Dashboard
```

本地模式优先默认值：

```text
工作区名称：我的本地工作区
专题名称：默认知识库 / Inbox / 个人研究
模型提供者：优先检测 Ollama，其次 LM Studio
```

涉及文件：

```text
web/src/app/(main)/setup/page.tsx
web/src/stores/topic-store.ts
web/src/lib/api.ts
```

验收：

1. 新用户首次进入可以完整完成设置；
2. 创建工作区后至少有一个专题；
3. Dashboard 能展示下一步导入入口；
4. 本地模型不可用时允许稍后配置。

---

### A3. 全局隐私与运行状态条

问题：

文档要求用户清楚知道数据、文件、模型、同步状态，目前信息分散。

方案：

新增 `PrivacyRuntimeBanner` 或 `WorkspaceStatusPanel`。

展示字段：

1. 当前工作区模式；
2. 数据库位置：本地 SQLite / 云端 PostgreSQL；
3. 文件位置：本地 Vault / S3；
4. 模型提供者：Ollama / LM Studio / OpenAI 等；
5. 是否会发送内容到云端模型；
6. 云端 Inbox 是否启用；
7. 云同步是否启用；
8. 最近一次 Runtime 检查状态。

推荐放置：

```text
Dashboard 顶部
设置首页
主布局顶部轻量状态入口
```

涉及文件：

```text
web/src/app/(main)/dashboard/page.tsx
web/src/app/(main)/settings/page.tsx
web/src/components/
web/src/lib/api.ts
```

验收：

1. 用户一眼知道数据是否留在本机；
2. 用户一眼知道模型是否在本地运行；
3. 用户一眼知道云端 Inbox 是否开启；
4. 点击可进入详细配置页。

## 阶段 B：云端 Inbox 与混合模式闭环

目标：实现文档 V0.2 的“手机端 -> 云端 Inbox -> 桌面端拉取 -> 本地库”。

### B1. Cloud Inbox API 接入

问题：

当前云端 Inbox 页面是模拟状态。

方案：

新增 `cloudInboxApi`：

```ts
cloudInboxApi.getSettings()
cloudInboxApi.updateSettings()
cloudInboxApi.getStatus()
cloudInboxApi.listRemoteItems()
cloudInboxApi.pullNow()
cloudInboxApi.getSyncLogs()
cloudInboxApi.disconnect()
```

设置项：

```ts
CloudInboxSettings {
  enabled: boolean
  pullStrategy: "manual" | "onStartup" | "scheduled"
  retention: "keep" | "deleteOriginal" | "deleteAll"
  cloudWorkspaceId?: string
  lastPulledAt?: string
}
```

涉及文件：

```text
web/src/app/(main)/settings/cloud-inbox/page.tsx
web/src/lib/api.ts

```

验收：

1. 开关状态可持久化；
2. 刷新页面后状态不丢失；
3. 能看到远端待拉取数量；
4. 手动拉取调用真实 API；
5. 拉取失败能展示错误原因；
6. 保留策略能被提交到后端。

### B3. 混合模式数据流可视化

实现状态：已完成。

问题：

混合模式目前只是一个模式值，用户看不到数据如何流动。

方案：

在混合工作区增加数据流卡片：

```text
手机采集
  -> 云端 Inbox
  -> 桌面拉取
  -> 本地 Inbox
  -> 本地处理
  -> 本地检索 / 问答 / MCP
```

每一段显示状态：

```text
未配置
已连接
同步中
失败
已关闭
```

验收：

1. 混合模式 Dashboard 能显示数据链路；
2. 关闭云端 Inbox 后链路中断状态清晰；
3. 本地处理仍可继续使用。

已落地：

1. 新增 `web/src/components/hybrid-data-flow-panel.tsx`；
2. Dashboard 接入紧凑版数据流卡片；
3. 设置总览接入完整数据流卡片；

## 9. 里程碑建议

### M1：本地可信任闭环

包含：

1. 本地免登录；
2. Setup 分步向导；
3. 隐私与运行状态组件；
4. Inbox 类型状态补齐。
5. Cloud Inbox 设置真实 API；
6. 手机端 `/capture` 文本 / URL / 文件采集。

目标：

```text
用户可以不注册账号，在本机完成知识库创建、导入、检索、问答和导出。
```

---

### M2：手机采集最小闭环

包含：

1. `/capture` 移动采集页；
2. 文本 / URL 进入 Inbox；
3. 云端 Inbox 设置真实接入；
4. 桌面端手动拉取。

目标：

```text
手机端发送 URL / 文本，桌面端可以拉取并导入本地知识库。
```

---

### M3：混合模式体验闭环

包含：

1. 混合模式数据流可视化；
2. 同步日志；
3. 云端保留策略；
4. 图片 / 文件采集；
5. 拉取失败重试。

目标：

```text
用户能够清楚、安全地使用“本地主库 + 云端 Inbox”的混合模式。
```

---

### M4：桌面形态闭环

包含：

1. Tauri 壳；
2. 本地服务自动启动；
3. 系统目录选择；
4. 打开 Vault / 导出目录；
5. 本地 Runtime 可视化。

目标：

```text
产品从 Web 控制台变成真正的本地优先桌面应用。
```

## 10. 最终建议

前端后续不要优先堆更多管理页，而应集中补齐三个核心体验：

```text
1. 本地可信任：不登录也能跑完整本地闭环。
2. 手机可采集：手机端能随手把资料送进 Inbox。
3. 混合可解释：用户清楚知道资料从哪里来、到哪里去、是否上传云端。
```

这三个体验打通后，当前已有的导入、检索、问答、报告、导出、Agent 管理页面才能真正组成文档规划中的“双模式 AI 知识资产引擎”。
