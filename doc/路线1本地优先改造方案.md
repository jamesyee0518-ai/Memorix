# AI/产业研究个人知识资产引擎路线改造方案

版本：V0.1  
改造方向：从“云端 SaaS 型知识库架构”调整为“路线 1：本地优先个人 AI 记忆引擎”  
适用阶段：MVP 重构 / 产品定位修正 / 技术路线调整  

---

## 1. 改造结论

之前的开发方案在**产品能力**上接近路线 1，但在**技术架构**上更偏路线 3，即云端 SaaS 架构。

原方案的核心能力包括：

```text
资料导入
  ↓
文档解析
  ↓
AI 摘要
  ↓
标签 / 实体 / 结构化
  ↓
分块与向量化
  ↓
混合检索
  ↓
RAG 问答
  ↓
报告导出
  ↓
Agent API
```

这些能力本身是正确的，可以保留。

但原方案默认使用：

```text
Web 前端
ASP.NET Core 后端
PostgreSQL / pgvector
Redis
MinIO
JWT 登录
API Key
云端 Worker
多用户隔离
```

这更适合 SaaS，而不是本地优先产品。

本次改造目标是将技术底座调整为：

```text
Tauri 桌面端
SQLite 本地数据库
本地 Vault 文件夹
本地任务队列
本地向量索引
Ollama / LM Studio / 用户自带 API Key
本地 MCP Server
Markdown / Obsidian 兼容导出
```

一句话结论：

> 保留原方案的“知识加工能力”，替换原方案的“SaaS 技术底座”，改造成本地优先、隐私优先、可被 Agent 调用的个人 AI 记忆引擎。

---

## 2. 新产品定位

### 2.1 原定位

原定位更接近：

> 面向专业用户的云端知识资产平台。

该定位的问题：

1. 一开始就需要账号系统、云端数据库、文件对象存储和权限系统；
2. 成本高，部署复杂；
3. 隐私信任门槛高；
4. 容易与 Notion、飞书、IMA、语雀、各类知识库产品正面竞争；
5. 个人开发者起步压力较大。

### 2.2 改造后定位

建议改为：

> 本地优先的个人 AI 记忆引擎。

更完整的表达：

> 面向研究者、创作者和开发者的本地优先 AI 知识资产工具，把网页、PDF、截图、录音、笔记和项目资料变成可搜索、可问答、可导出、可被 Agent 调用的长期记忆库。

### 2.3 产品关键词

```text
本地优先
隐私安全
个人长期记忆
多模态采集
AI 清洗
Markdown 开放格式
Obsidian 兼容
混合检索
RAG 问答
MCP 接入
可选云增强
```

---

## 3. 路线调整原则

本次改造遵循三个原则：

### 3.1 路线 1 为主

以本地优先工具作为 MVP 主线。

```text
本地存储
本地索引
本地任务
本地模型优先
本地 MCP Server
```

### 3.2 路线 2 兼容

吸收 Obsidian 的开放文件优势。

```text
Markdown 文件
Vault 目录
双链兼容
本地文件夹可迁移
Obsidian 导出
```

但不做完整 Obsidian 替代品，不陷入编辑器、插件、主题生态泥潭。

### 3.3 路线 3 后置

云端 SaaS 不作为第一阶段主线。

后续可以逐步加入：

```text
可选云同步
移动端采集
团队空间
私有部署
SaaS 订阅
模板市场
```

## 4. 架构改造总览

### 4.1 原架构

```text
Web 前端
  ↓
API 服务
  ↓
PostgreSQL / pgvector
Redis
MinIO
Worker Service
  ↓
云端大模型 / 本地模型
```

### 4.2 改造后架构

```text
Tauri Desktop App
  ↓
Local Core Service
  ↓
SQLite 本地数据库
本地 Vault 文件夹
本地任务队列
本地向量索引
  ↓
AI Provider Layer
  ├── Ollama
  ├── LM Studio
  ├── 用户自带 OpenAI / Claude / Gemini API Key
  └── 后续可选云端托管模型
  ↓
RAG / Memory Engine
  ↓
Local MCP Server / localhost API
  ↓
Claude / Cursor / Hermes / 本地 Agent
```

### 4.3 改造后核心特征

| 维度 | 改造前 | 改造后 |
|---|---|---|
| 产品形态 | Web SaaS | 桌面应用 |
| 数据库 | PostgreSQL | SQLite |
| 文件存储 | MinIO / S3 | 本地 Vault 文件夹 |
| 用户体系 | 注册登录 / JWT | 本地 Profile / 本地设置 |
| 向量检索 | pgvector | SQLite 向量扩展 / 本地向量索引 |
| 任务队列 | Redis / Worker | 本地 jobs 表 + 本地 Worker |
| Agent 接入 | 云端 Agent API | 本地 MCP Server |
| 模型调用 | 云端模型优先 | 本地模型优先，可选云模型 |
| 隐私 | 数据上传云端 | 数据默认留在本地 |
| 起步复杂度 | 高 | 中低 |

## 5. 技术栈改造建议

## 5.1 桌面端

推荐：

```text
Tauri + React / Vue + TypeScript
```

原因：

1. 打包体积小；
2. 本地文件权限支持好；
3. 可以调用 Rust 侧能力；
4. 比 Electron 更轻；
5. 适合个人知识桌面工具。

备选：

```text
Electron
```

不建议作为第一选择，除非团队更熟悉 Electron。

---

## 5.2 本地数据库

推荐：

```text
SQLite
```

用于保存：

1. sources；
2. documents；
3. chunks；
4. tags；
5. entities；
6. relations；
7. jobs；
8. reports；
9. settings；
10. model_configs。

SQLite 的优势：

1. 单文件部署；
2. 无需数据库服务；
3. 易备份；
4. 适合个人本地工具；
5. 可随 Vault 一起迁移。

---

## 5.3 本地文件层

推荐采用 Vault 目录：

```text
~/KnowledgeMemory/Vault/
  sources/
  documents/
  attachments/
  reports/
  exports/
  snapshots/
  indexes/
```

### 5.3.1 双轨存储原则

```text
SQLite：结构化索引、状态、向量、任务、关系
本地文件夹：原始文件、Markdown、附件、导出文件
```

不要只用 SQLite，也不要只用 Markdown。

| 方案 | 问题 |
|---|---|
| 只用 SQLite | 用户不放心，迁移性弱 |
| 只用 Markdown | 状态、向量、任务、关系难维护 |
| SQLite + Vault | 最平衡 |

---

## 5.4 本地 AI 接入

第一阶段优先支持：

```text
Ollama
LM Studio
用户自带 OpenAI-compatible API
```

模型配置统一抽象成 Provider：

```text
AIProvider
  ├── OllamaProvider
  ├── LMStudioProvider
  ├── OpenAICompatibleProvider
  └── CloudProvider，后续
```

### 5.4.1 模型用途分类

| 模型类型 | 用途 |
|---|---|
| Chat Model | 摘要、问答、报告 |
| Embedding Model | 向量化 |
| Vision Model | 截图 / 图片理解，后续 |
| ASR Model | 录音转写，后续 |

---

## 5.5 本地向量索引

可选方案：

| 方案 | 说明 |
|---|---|
| sqlite-vec | SQLite 内向量扩展，适合本地工具 |
| sqlite-vss | SQLite 向量扩展，视维护情况选择 |
| LanceDB | 本地向量库，适合文件化存储 |
| Qdrant local | 功能强，但引入服务复杂度 |
| 内存 HNSW + 持久化 | 可后续自研轻量方案 |

MVP 建议优先评估：

```text
SQLite + sqlite-vec
```

如果实现困难，可以先采用：

```text
SQLite 保存 chunk
向量索引用本地文件 / LanceDB
```

## 7. 新本地数据模型

## 7.1 profiles：本地用户配置

替代原 `users` 表。

```sql
CREATE TABLE profiles (
    id TEXT PRIMARY KEY,
    name TEXT,
    avatar_path TEXT,
    vault_path TEXT NOT NULL,
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

说明：

1. 第一版通常只有一个 profile；
2. 后续可支持多个工作区；
3. 不需要 password_hash；
4. 不需要 email；
5. 不需要 JWT。

---

## 7.2 settings：本地设置

```sql
CREATE TABLE settings (
    key TEXT PRIMARY KEY,
    value TEXT NOT NULL,
    updated_at TEXT NOT NULL
);
```

用于保存：

1. 默认模型；
2. 默认 embedding 模型；
3. Vault 路径；
4. 是否启用 MCP；
5. 是否启用云模型；
6. UI 设置。

```sql
CREATE TABLE model_configs (

    name TEXT NOT NULL,
    base_url TEXT,

    model_name TEXT NOT NULL,
    model_type TEXT NOT NULL,
```

# 20. 改造风险与应对

## 20.1 Tauri 本地能力复杂

风险：

1. 文件权限；
2. 打包签名；
3. 跨平台差异；
4. Rust 与前端通信。

应对：

1. 第一版只支持 macOS；
2. Windows 后续适配；
3. 先跑通核心链路，不追求复杂系统集成。

---

## 20.2 本地模型环境复杂

风险：

1. 用户没有安装 Ollama；
2. 模型不存在；
3. 本地模型输出不稳定；
4. 性能差异大。

应对：

1. 支持 OpenAI-compatible API 作为备选；
2. 提供模型健康检查；
3. 提供推荐模型配置；
4. AI 功能失败时不影响资料保存。

---

## 20.3 向量库选择不确定

风险：

1. sqlite-vec 生态不成熟；
2. sqlite-vss 维护情况不确定；
3. LanceDB 额外引入复杂度。

应对：

1. 抽象 VectorStore 接口；
2. MVP 先做 FTS + 简单向量方案；
3. 后续可替换底层向量实现。

接口：

```text
VectorStore
  ├── upsert(chunk_id, embedding)
  ├── search(query_embedding, filters, top_k)
  ├── delete_by_document(document_id)
  └── rebuild(topic_id)
```

---

## 20.4 本地文件与数据库一致性

风险：

1. 用户手动移动文件；
2. SQLite 记录和文件路径失配；
3. 导出文件被删除。

应对：

1. 定期完整性检查；
2. 文件 hash 校验；
3. 缺失文件标记 warning；
4. 提供重建索引能力。

---

## 20.5 过早做云同步

风险：

1. 增加账号系统；
2. 增加合规压力；
3. 增加成本；
4. 破坏本地优先定位。

应对：

1. 云同步放到阶段六；
2. 第一版只做本地；
3. 导出和备份先解决迁移问题。

# 24. 最终建议

本次改造不是推翻原方案，而是把原方案从 SaaS 技术路线调整为本地优先路线。

```text

AI 摘要

分块向量

```

应替换：

```text
Web SaaS
PostgreSQL
MinIO
Redis
JWT
API Key
云端 Worker
```

替换为：

```text
Tauri
SQLite
本地 Vault
本地 jobs
本地模型
本地向量索引
MCP Server
```

最终落点：

> 先做一个真正可用的本地优先个人 AI 记忆引擎，再考虑云同步、团队协作和 SaaS 商业化。

```text

  ↓
本地检索
  ↓

  ↓
```

只要这个闭环成立，后续做云端、团队、市场才有基础。
