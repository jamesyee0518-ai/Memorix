# 双模式知识引擎  
# 阶段一：双模式底座改造详细设计 V0.1

## 1. 阶段定位

阶段一的目标不是做完整知识库，也不是立刻实现手机端、RAG、报告、MCP。

阶段一只解决一个底层问题：

> **让同一套产品能力，未来可以同时运行在本地模式和云端模式下。**

也就是说，阶段一要把系统从“单一 SaaS 架构”或“单一本地架构”，改造成：

```text
统一应用层
  ↓
Workspace Mode Router
  ↓
Local Runtime / Cloud Runtime
  ↓
统一业务模型
```

阶段一的核心不是堆功能，而是完成底座抽象：

```text
Workspace 抽象
Storage 抽象
File 抽象
Model 抽象
Job 抽象
Config 抽象
Runtime Router 抽象
```

如果阶段一没有做好，后续本地、云端、手机端、同步、MCP 都会变成重复开发。

---

## 2. 阶段一目标

阶段一完成后，系统必须具备以下能力：

1. 支持创建本地工作区；
2. 支持创建云端工作区的结构预留；
3. 支持工作区模式识别：local / cloud / hybrid；
4. 本地模式可以初始化 SQLite；
5. 本地模式可以初始化本地 Vault 文件夹；
6. 云端模式保留 PostgreSQL / S3 / Worker 接口；
7. 业务层不直接依赖 SQLite 或 PostgreSQL；
8. 文件读写不直接依赖本地路径或 S3；
9. 模型调用不直接依赖某一家模型服务；
10. 任务队列不直接依赖 Redis 或本地队列表；
11. 前端可以展示当前工作区模式；
12. 用户可以在启动时选择使用模式；
13. 系统可以根据 workspace.mode 路由到不同 Runtime；
14. 后续资料导入、AI 摘要、检索、RAG 可以复用同一套业务接口。

---

## 3. 阶段一不做的内容

阶段一明确不做：

1. 不做完整资料导入；
2. 不做 AI 摘要；
3. 不做向量检索；
4. 不做 RAG 问答；
5. 不做报告生成；
6. 不做手机端采集；
7. 不做本地云端双向同步；
8. 不做完整用户计费；
9. 不做团队协作；
10. 不做插件市场；
11. 不做复杂权限系统；
12. 不做完整 Agent API。

阶段一只做底座。

直接判断：

> 如果阶段一开始就做手机端、RAG、同步，会导致底层抽象不稳，后面返工成本很高。

## 4. 双模式总体架构

### 4.1 最终目标架构

```text
Desktop App / Web App / Mobile App
          ↓
     Application Layer
          ↓
   Workspace Mode Router
          ↓
 ┌─────────────────────┬─────────────────────┐
 │ Local Runtime        │ Cloud Runtime        │
 │                     │                     │
 │ SQLite              │ PostgreSQL           │
 │ Local Vault          │ S3 / MinIO           │
 │ Local Job Queue      │ Redis / Worker       │
 │ Ollama / LM Studio   │ Cloud Model API      │
 │ Local MCP Server     │ Cloud Agent API      │
 └─────────────────────┴─────────────────────┘
```

### 4.2 阶段一最小架构

阶段一只需要做到：

```text
Desktop / Web Shell
      ↓
Workspace Manager
      ↓
Runtime Router
      ↓
Local Runtime 可运行
Cloud Runtime 接口预留
```

也就是说，阶段一可以先实现 Local Runtime，Cloud Runtime 先保留接口。

但代码结构必须支持：

```text

LocalJobQueue

```

## 5. 核心设计原则

### 5.1 业务模型统一

本地模式和云端模式必须共用核心业务模型。

统一模型包括：

```text
workspace
topic
inbox_item
source
document
chunk
tag
entity
report
job
setting
```

不同的是存储实现：

```text
local  → SQLite + Local Vault
cloud  → PostgreSQL + S3 / MinIO
hybrid → Local 主库 + Cloud Inbox / Sync
```

### 5.2 业务层不能直接依赖具体数据库

错误写法：

```ts
const docs = await sqlite.query(...)
```

或者：

```csharp
_dbContext.Documents.Add(...)
```

直接散落在业务代码里。

正确做法：

```ts
await knowledgeRepository.createDocument(...)
```

然后由 Runtime 决定走：

```text
LocalKnowledgeRepository
CloudKnowledgeRepository
```

### 5.3 文件系统必须抽象

本地模式：

```text
~/KnowledgeEngine/Vault/
```

云端模式：

```text
S3 / MinIO Bucket
```

业务代码不能直接写：

```ts
fs.writeFile(...)
```

或直接写：

```ts
s3.putObject(...)
```

而应该调用：

```ts
fileStorage.saveFile(...)
```

### 5.4 模型服务必须抽象

本地模型：

```text
Ollama
LM Studio
llama.cpp server
```

云端模型：

```text
OpenAI
Anthropic
Gemini
DeepSeek
自定义 OpenAI-compatible API
```

业务层只调用：

```ts
modelProvider.chat(...)
modelProvider.embed(...)
```

### 5.5 手机端不直接写本地库

手机端未来第一版应写入：

```text
Cloud Inbox
```

或者：

```text
LAN Local Server
```

不要让手机端直接操作本地 SQLite。

## 7. Workspace 模块设计

### 7.1 Workspace 是什么

Workspace 是用户知识库的运行单位。

一个 Workspace 代表：

1. 一套资料库；
2. 一套路由模式；
3. 一套存储配置；
4. 一套文件配置；
5. 一套模型配置；
6. 一套任务队列；
7. 一套后续同步策略。

### 7.2 Workspace 类型

| 模式 | 说明 |
|---|---|
| local | 数据主要保存在本机 |
| cloud | 数据保存在云端 |
| hybrid | 本地为主，云端用于 Inbox / 同步 / 备份 |

阶段一建议先实现：

```text
local
cloud-placeholder
```

hybrid 先做字段预留。

### 7.3 Workspace 数据结构

```ts
type WorkspaceMode = "local" | "cloud" | "hybrid";

interface Workspace {
  id: string;
  name: string;
  mode: WorkspaceMode;

  storageProvider: "sqlite" | "postgres";
  fileProvider: "local_fs" | "s3" | "minio";
  jobProvider: "local_queue" | "redis";
  modelProvider: "ollama" | "lmstudio" | "openai" | "anthropic" | "custom";

  localDbPath?: string;
  localVaultPath?: string;

  cloudApiBaseUrl?: string;
  cloudWorkspaceId?: string;

  syncEnabled: boolean;
  inboxEnabled: boolean;

  createdAt: string;
  updatedAt: string;
}
```

```text

  ↓
```

## 10. Cloud Runtime 设计

### 10.1 Cloud Runtime 阶段一目标

阶段一不需要完整云端 SaaS，但要保留接口。

Cloud Runtime 最终负责：

1. 连接云端 API；
2. 使用云端 PostgreSQL；
3. 使用 S3 / MinIO；
4. 使用云端 Worker；
5. 使用云端模型；
6. 支持手机端 Inbox；
7. 支持多设备同步。

### 10.2 Cloud Runtime 阶段一最小实现

阶段一只需要：

1. 定义 Cloud Runtime 接口；
2. 定义 Cloud Repository 接口；
3. 定义 Cloud FileStorage 接口；
4. 定义 Cloud ModelProvider 接口；
5. 前端保留云端模式入口；
6. 可以配置 cloudApiBaseUrl；
7. 不强制跑通完整云端。

## 11. Runtime Router 设计

### 11.1 Runtime Router 作用

Runtime Router 根据当前 workspace.mode 选择具体实现。

```text
Application Service
      ↓
Runtime Router
      ↓
Local Runtime / Cloud Runtime
```

### 11.2 Runtime Router 示例

```ts
class RuntimeRouter {
  constructor(private workspaceManager: WorkspaceManager) {}

  getRepository(): KnowledgeRepository {
    const workspace = this.workspaceManager.current();

    if (workspace.mode === "local") {
      return new LocalKnowledgeRepository(workspace.localDbPath);
    }

    if (workspace.mode === "cloud") {
      return new CloudKnowledgeRepository(workspace.cloudApiBaseUrl);
    }

    if (workspace.mode === "hybrid") {
      return new HybridKnowledgeRepository(workspace);
    }

    throw new Error("Unsupported workspace mode");
  }

  getFileStorage(): FileStorage {
    const workspace = this.workspaceManager.current();

    if (workspace.fileProvider === "local_fs") {
      return new LocalFileStorage(workspace.localVaultPath);
    }

    if (workspace.fileProvider === "s3") {
      return new S3FileStorage(workspace);
    }

    throw new Error("Unsupported file provider");
  }
}
```

## 12. Repository 抽象设计

### 12.1 KnowledgeRepository 接口

```ts
interface KnowledgeRepository {
  createTopic(input: CreateTopicInput): Promise<Topic>;
  getTopic(id: string): Promise<Topic | null>;
  listTopics(): Promise<Topic[]>;

  createInboxItem(input: CreateInboxItemInput): Promise<InboxItem>;

  createSource(input: CreateSourceInput): Promise<Source>;
  getSource(id: string): Promise<Source | null>;
  listSources(filter: SourceFilter): Promise<Source[]>;

  createDocument(input: CreateDocumentInput): Promise<Document>;
  getDocument(id: string): Promise<Document | null>;
  listDocuments(filter: DocumentFilter): Promise<Document[]>;

  saveChunks(documentId: string, chunks: DocumentChunk[]): Promise<void>;
  searchDocuments(input: SearchInput): Promise<SearchResult[]>;

  createJob(input: CreateJobInput): Promise<Job>;
  getNextPendingJob(): Promise<Job | null>;
  updateJobStatus(id: string, status: JobStatus): Promise<void>;
}
```

### 12.2 本地实现

```text
LocalKnowledgeRepository
  ↓
SQLite
```

本地实现直接操作 SQLite。

### 12.3 云端实现

```text
CloudKnowledgeRepository
  ↓
HTTP API
  ↓
Cloud Backend
  ↓
PostgreSQL
```

云端实现可以先是 HTTP Client，不一定直接连 PostgreSQL。

### 12.4 为什么云端实现不建议直接连 PostgreSQL

桌面端或手机端不应该直连云数据库。

正确方式：

```text
Client
  ↓
Cloud API
  ↓
Cloud Backend
  ↓
PostgreSQL
```

这样才能做：

1. 鉴权；
2. 权限；
3. 限流；
4. 审计；
5. 版本兼容；
6. 安全控制。

## 24. 阶段一关键风险

### 24.1 抽象过度

风险：

一开始设计太多接口和模式，导致迟迟无法跑通产品。

应对：

只抽象 5 个核心接口：

```text
Repository
FileStorage
ModelProvider
JobQueue
RuntimeRouter
```

其他先不要复杂化。

### 24.2 抽象不足

风险：

直接写死 SQLite、本地路径、Ollama，后续接云端时大面积重构。

应对：

阶段一必须完成 Runtime Router。

### 24.3 本地和云端模型割裂

风险：

本地一套表，云端一套表，后续同步困难。

应对：

核心业务模型必须统一。

### 24.4 手机端提前进入

风险：

还没有 Inbox / Workspace / Cloud Runtime，就先做手机端，会导致数据流混乱。

应对：

手机端放到后续阶段，阶段一只预留 inbox_items 表。

### 24.5 同步提前复杂化

风险：

一开始做双向同步会极大增加复杂度。

应对：

阶段一只预留 sync_enabled 字段，不做同步逻辑。

## 27. 阶段一总结

阶段一的本质是：

> **把项目从单一架构改造成可同时承载本地和云端的双模式底座。**

它的重点不是功能数量，而是架构边界：

```text
Workspace 决定运行模式
Runtime Router 决定具体实现
Repository 隔离数据库
FileStorage 隔离文件系统
ModelProvider 隔离模型服务
JobQueue 隔离任务队列
```

阶段一是否成功，只看 5 个问题：

1. 本地 Workspace 能不能完整初始化？
2. 业务层是否不再直接绑定 SQLite / PostgreSQL？
3. 文件层是否不再直接绑定本地 FS / S3？
4. 模型层是否可以自由切换本地和云端？
5. 后续阶段能否在不重写业务逻辑的情况下接入 Cloud Runtime？

只要这 5 点成立，阶段一就是合格的。
